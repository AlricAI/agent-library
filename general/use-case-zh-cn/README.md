## 课程管理

## Use Case ID：UC－SCHED－02

## Use Case Name

教务创建课程并把教师／学生 enroll 到固定课表（含系统智能建议以减少跨校区）

## Primary Actor

－教务

## Supporting Actors

- 学导（提供需求与确认）
- 系统（课程创建模块、排课引擎／建议引擎、日程／资源管理、通知中心）
- 教师（被分配）
- 学生（被enroll）

## Description

－教务基于学导提交的排课需求或其他来源，在系统中创建课程（在创建时确定固定上课时间与周期），并将教师与学生 enroll 到该已创建课程；系统在创建／分配阶段提供智能建议以优化教师排班、减少教师跑校区次数和合理分配科目频率。

## Trigger

－教务查看到学导提交的排课需求，或主动创建新课程以接收学生报名／排入。

## Preconditions

- 系统中已同步教师日程、校区教室可用性与学生可用时间窗。
- 规则引擎配置了约束（教师不可重叠、最大学生容量、校区切换限制等）。
- 教务拥有创建课程与enroll权限。

## Main Flow

1．教务在＂创建课程＂页选择或关联排课需求（可新建课程也可用模板）。
2．在创建流程中，教务选择课程基本信息（课程名／科目／级别／总课时／每课时时长／周期与固定上课时间段／上课校区／教室规格／课程容量／候补策略）。

3．系统基于已同步的教师可用性、校区教师分布、学生时间偏好及完成期限，自动生成若干（通常 3 个）分配建议方案，分别展示：教师匹配、跨校区成本、教师跑校次数估算、学生便利度评分与教务负担指标。

4．教务查看建议方案并进行微调（替换教师、调整教室或修改容量），或选择＂系统自动分配并执行＂。

5．教务确认后，系统将所选教师与目标学生enroll到已创建课程：系统在教师和学生日程中占座并在学生个人课表显示该课程。

6．系统发送通知给教师、学生和学导（课程已创建／已被enroll／课程时间与地点）。
7．系统更新资源状态（教师忙碌表、教室占用）并在数据仓库记录为可统计指标（跑校次数、教师负载等）。

## Alternative Flow

－A1：若部分学生时间不匹配，教务可选择创建多个平行班次或向学生推送可选时间以确认再enroll。
－A2：教务先锁定课程时间与容量，允许学生按报名先后自动enroll（按规则）或由教务手动enroll关键学生。

## Exception Flow

－E1：在确认enroll时检测到教师时间冲突（并发分配）→ 系统提示冲突并建议替代教师或时间，若无法自动解决，申请进入人工调度。
－E2：教室资源在确认后被占用或被取消 → 系统回滚该占用并提示教务重新选择教室或修改时间。
－E3：学生在enroll期间付款或合同状态异常 → 系统将该学生暂缓enroll并通知教务与学导处理。

## Postconditions

- 课程被创建并在学生课表／教师日历中可见。
- 资源占用（教师／教室／学生名额）被锁定并记录。
- 排课相关的统计数据（教师跨校次数、班级密度、学生完成期限匹配度）被写入运营报表。

## Business Rules

－创建课程时必须先确定固定上课时间与周期，教务仅能在该框架内enroll教师与学生或拆分为平行班次。

- 教师不可被分配到时间冲突的课程（系统强制校验）。
- 系统在生成建议时优先考虑：教师同校区匹配 → 最小化跨校区次数 → 教师评价与可用性 → 学生便利度。
  －若学生为高优先级或合同约束需在指定期限内完成课程，可提升该学生在enroll顺序中的优先级。
  －所有自动分配或教务确认操作应产生日志以便回滚与审计。

## Notes（实现／产品建议）

－建议将＂建议引擎＂作为可选层：先提供自动建议供教务决策，默认不强制自动执行，降低误判风险。
－在教务确认页面提供＂影响预估＂面板（展示教师跑校变化、成本、学生满意度预估），帮助快速决策。
－支持批量enroll（一次性将一组学生加入课程）与按规则自动入班（例如按照报名时间／优先级自动入座）。
－将课程模板与标准化
