## 课程管理

## Use Case ID：UC－SCHED－02

## Use Case Name

教务创建课程并把教师／学生 enroll 到固定课表（含系统智能建议以减少跨校区）

## Primary Actor

－教务

## Supporting Actors

- 学导（提供需求与确认）
- 系统（课程创建模块、排课引擎／建议引擎、日程／资源管理、通知中心）
- 教师（被分配）
- 学生（被enroll）

## Description

－教务基于学导提交的排课需求或其他来源，在系统中创建课程（在创建时确定固定上课时间与周期），并将教师与学生 enroll 到该已创建课程；系统在创建／分配阶段提供智能建议以优化教师排班、减少教师跑校区次数和合理分配科目频率。

## Trigger

－教务查看到学导提交的排课需求，或主动创建新课程以接收学生报名／排入。

## Preconditions

- 系统中已同步教师日程、校区教室可用性与学生可用时间窗。
- 规则引擎配置了约束（教师不可重叠、最大学生容量、校区切换限制等）。
- 教务拥有创建课程与enroll权限。

## Main Flow

1．教务在＂创建课程＂页选择或关联排课需求（可新建课程也可用模板）。
2．在创建流程中，教务选择课程基本信息（课程名／科目／级别／总课时／每课时时长／周期与固定上课时间段／上课校区／教室规格／课程容量／候补策略）。

3．系统基于已同步的教师可用性、校区教师分布、学生时间偏好及完成期限，自动生成若干（通常 3 个）分配建议方案，分别展示：教师匹配、跨校区成本、教师跑校次数估算、学生便利度评分与教务负担指标。

4．教务查看建议方案并进行微调（替换教师、调整教室或修改容量），或选择＂系统自动分配并执行＂。

5．教务确认后，系统将所选教师与目标学生enroll到已创建课程：系统在教师和学生日程中占座并在学生个人课表显示该课程。

6．系统发送通知给教师、学生和学导（课程已创建／已被enroll／课程时间与地点）。
7．系统更新资源状态（教师忙碌表、教室占用）并在数据仓库记录为可统计指标（跑校次数、教师负载等）。

## Alternative Flow

－A1：若部分学生时间不匹配，教务可选择创建多个平行班次或向学生推送可选时间以确认再enroll。
－A2：教务先锁定课程时间与容量，允许学生按报名先后自动enroll（按规则）或由教务手动enroll关键学生。

## Exception Flow

－E1：在确认enroll时检测到教师时间冲突（并发分配）→ 系统提示冲突并建议替代教师或时间，若无法自动解决，申请进入人工调度。
－E2：教室资源在确认后被占用或被取消 → 系统回滚该占用并提示教务重新选择教室或修改时间。
－E3：学生在enroll期间付款或合同状态异常 → 系统将该学生暂缓enroll并通知教务与学导处理。

## Postconditions

- 课程被创建并在学生课表／教师日历中可见。
- 资源占用（教师／教室／学生名额）被锁定并记录。
- 排课相关的统计数据（教师跨校次数、班级密度、学生完成期限匹配度）被写入运营报表。

## Business Rules

－创建课程时必须先确定固定上课时间与周期，教务仅能在该框架内enroll教师与学生或拆分为平行班次。

- 教师不可被分配到时间冲突的课程（系统强制校验）。
- 系统在生成建议时优先考虑：教师同校区匹配 → 最小化跨校区次数 → 教师评价与可用性 → 学生便利度。
  －若学生为高优先级或合同约束需在指定期限内完成课程，可提升该学生在enroll顺序中的优先级。
  －所有自动分配或教务确认操作应产生日志以便回滚与审计。

## Notes（实现／产品建议）

－建议将＂建议引擎＂作为可选层：先提供自动建议供教务决策，默认不强制自动执行，降低误判风险。
－在教务确认页面提供＂影响预估＂面板（展示教师跑校变化、成本、学生满意度预估），帮助快速决策。
－支持批量enroll（一次性将一组学生加入课程）与按规则自动入班（例如按照报名时间／优先级自动入座）。
－将课程模板与标准化时间槽结合，便于快速创建重复课程并减少人工配置量。

## 用户基础信息管理：开发难度低

## Use Case ID：UC－INFO－01

## Use Case Name

按角色权限查询学生与家长联系方式

## Primary Actor

- 顾问
- 学导

## Supporting Actors

- 系统（用户管理与权限模块、数据安全模块）
- 学生
- 家长

## Description

－顾问和学导可在系统内查询其负责学生及对应家长的联系方式，用于日常沟通与紧急联系；系统按数据权限进行访问控制与审计。

## Trigger

－顾问或学导进入学生详情页或联系人列表页面。

## Preconditions

- 学生档案中已录入联系方式与家长信息。
- 用户已登录系统且具备对应学生的归属权限。
- 权限规则已配置（仅可查看＂自己负责＂的学生）。

## Main Flow

1．顾问或学导进入＂我的学生＂列表。
2．系统根据账号权限加载其负责的学生名单。
3．用户点击某学生进入详情页。
4．系统展示学生联系方式（电话／微信／邮箱等）及家长联系方式。
5．用户可点击拨号／复制联系方式进行沟通。
6．系统记录该联系方式访问行为（时间、用户、学生ID）。

## Alternative Flow

- A1：用户通过搜索框查找学生（在权限范围内）。
- A2：系统支持批量导出联系方式（如允许），并记录导出日志。

## Exception Flow

- E1：用户尝试访问非本人负责学生 → 系统拒绝访问并提示无权限。
- E2：学生信息不完整 → 系统提示＂联系方式未录入＂。

## Postconditions

- 用户成功查看联系方式。
- 系统生成访问日志供审计。

## Business Rules

- 顾问／学导仅能查看其名下或分配范围内学生信息。
- 联系方式属于敏感信息，需记录访问日志。
- 禁止跨校区或跨权限访问学生数据（除非具备高级权限）。
- 支持数据脱敏显示（例如部分号码隐藏）以增强安全性（可配置）。

## Use Case ID：UC－INFO－02

## Use Case Name

按角色权限查询教师联系方式

## Primary Actor

- 教务
- 教学负责人

## Supporting Actors

- 系统（用户管理与权限模块、数据安全模块）
- 教师

## Description

－教务与教学负责人可在系统内查询教师联系方式，用于课程安排、迟到、突发状况等紧急沟通。

## Trigger

－教务或教学负责人进入教师详情页或教师列表页面。

## Preconditions

- 教师档案中已录入联系方式。
- 用户具备教师信息查询权限。

## Main Flow

1．教务或教学负责人进入＂教师管理＂页面。
2．系统加载可查看的教师列表（按校区或权限范围）。
3．用户点击教师姓名进入详情页。

4．系统展示教师联系方式（电话、邮箱、紧急联系人等）。
5．用户可直接拨号或复制联系方式。
6．系统记录访问行为日志。

## Alternative Flow

- A1：用户通过课程详情页直接跳转查看任课教师联系方式。
- A2：紧急模式下提供＂一键联系教师＂功能（如电话直拨）。

## Exception Flow

- E1：用户权限不足 → 系统拒绝访问并提示无权限。
- E2：教师联系方式缺失 → 系统提示资料不完整。

## Postconditions

- 教师联系方式被成功查询。
- 系统记录访问日志。

## Business Rules

- 教务可查看本校区教师信息；教学负责人可查看所属管理范围内教师。
- 联系方式访问必须记录日志以符合数据合规要求。
- 禁止非授权角色查看教师私人联系方式。
- 支持紧急联系场景下的优先展示（如当前授课教师）。

## Notes（实现／产品建议）

- 建议加入数据访问审计报表，便于内部合规检查。
- 支持联系方式字段分级管理（办公电话／私人电话分权限展示）。
- 提供＂最近授课教师快速联系入口＂提升应急处理效率。

## 班级信息管理：开发难度中等

## Use Case ID：UC－CLASS－01

## Use Case Name

查看班级学生名单与学生档案

## Primary Actor

－教师（或班主任）

## Supporting Actors

- 学导
- 教务
- 系统（学生信息模块、权限模块、通知中心）

## Description

－授课教师或相关角色在系统中查看所属班级的学生名单与单个学生的详细档案（基本信息、联系方式、出勤记录、作业／成绩摘要、课时余额与历史变更记录），以便教学管理与个性化跟进。

## Trigger

－教师进入＂班级管理＂或点击课程页面的＂学生名单／学生档案＂链接。

## Preconditions

- 教师已登录且具备查看该班级／学生信息的权限。
- 学生档案已在系统中建立并包含必要字段（联系方式、合同／课时数据、出勤记录等）。

## Main Flow

1．教师进入＂班级管理＂页面并选择指定班级。
2．系统展示班级基本信息与当前学生名单（姓名、学号／标签、出勤／欠费／警示标识等摘要字段）。

3．教师可按姓名／学号／状态筛选或排序学生列表（例如按出勤率、课时余额、红色预警优先）。

4．教师点击某学生姓名进入该学生档案页。
5．系统展示学生档案：基本信息、家长联系方式、合同与课时余额、最近出勤记录、作业完成情况、学情分析摘要、历史变更与沟通记录。

6．教师可从档案页直接发起联系（拨号／发消息）、添加课堂备注、创建跟进任务或发起补课／点评操作。

7．系统保存教师的查看与操作日志以供审计。

## Alternative Flow

- A1：教师批量导出班级学生名单或生成班级报表（Excel／PDF）。
- A2：教师在名单视图中直接进行群发通知或布置作业，系统记录群发历史。
- A3：教师根据筛选结果创建关注列表（例如本周需跟进学生）并生成任务分配给学导。

## Exception Flow

－E1：教师尝试查看非所属班级或无权限学生 → 系统拒绝访问并提示无权限，并记录访问尝试。
－E2：学生档案信息不完整或数据同步延迟 → 系统标注＂数据不完整＂并提供补录入口或触发数据同步任务。
－E3：导出或批量操作失败（权限／网络／文件服务异常）→ 系统返回错误并提示重试或联系管理员。

## Postconditions

- 教师成功查看班级名单或学生档案。
- 任何对学生档案的修改或新增的教学备注被持久化并记录操作人／时间。
- 相关操作（拨打、消息、任务创建）生成待办或沟通记录供后续追踪。

## Business Rules

- 仅允许有权限的角色查看或导出敏感学生信息；所有访问需生成审计日志。
- 联系方式展示可按角色进行脱敏（例如仅显示后四位或需点击＂显示完整＂并记录行为）。
  －学生档案的关键字段（课时余额、合同状态、出勤）以系统主数据为准，禁用手动随意修改，需通过标准流程变更。
- 批量导出操作应受限并记录导出理由与接收人。
- 对于标记为＂敏感／保密＂或法律限制的学生（如特殊隐私），系统应进一步限制访问与导出权限。

## Notes（实现／产品建议）

－建议在学生名单页提供可视化列（出勤率仪表、课时剩余条形图、学情热度色带），提升教师快速识别能力。

- 在档案页加入＂最近沟通与行动＂卡片，便于教师快速了解历史跟进状态。
- 支持快速动作（拨号、一键发消息、建任务）以减少上下文切换。
- 将访问日志与数据权限管理整合到合规报表，便于定期审计。

## 课堂反馈与记录：开发难度中等偏高

Use Case ID：UC－FEED－01

## Use Case Name

课堂反馈与课堂记录（文字／拍照／视频）

## Primary Actor

－教师

## Supporting Actors

- 学生（被反馈对象）
- 学导／顾问（查看与跟进）
- 系统（课堂反馈模块、媒体存储、通知中心、审计／权限模块）

## Description

－课堂结束后教师在系统中提交课堂反馈，支持文字、拍照、视频等多媒体形式；反馈用于记录课堂表现、作业布置、问题点与后续跟进，并可被学导／顾问查看以执行教学干预或家校沟通。

## Trigger

－课堂结束或教师主动提交课堂反馈。

## Preconditions

- 该节课程已在系统中排期并标记为＂已上课＂或＂已结束＂。
- 教师已登录且拥有相应课程的反馈权限。
- 系统具备多媒体上传与存储能力，且有合规的存储策略（大小／时长／格式限制）。

## Main Flow

1．课堂结束后教师进入该节课详情或＂我的课堂＂→点击＂填写课堂反馈＂。
2．教师选择反馈类型并输入内容：文字说明（必填或选填）、上传照片（拍照或从相册选择）、上传／录制视频、可附加评分项（出勤、课堂参与、作业完成度等）。

3．教师可选择是否将反馈设为＂仅内部可见＂或＂允许家长查看＂。
4．教师提交反馈；系统进行媒体校验（格式、大小、时长）并保存多媒体到指定存储位置，生成反馈记录ID。

5．系统将反馈同步到学生档案与班级记录，并根据权限推送通知给学导／顾问／家长 （若允许）。

6．学导或顾问查看反馈并在必要时创建跟进任务（如安排补课、联系家长、布置额外练习）。

7．所有反馈条目与媒体有审计日志并可按课程／学生／教师维度检索。

## Alternative Flow

- A1：教师选择＂草稿保存＂稍后完善再提交。
- A2：教师在提交后编辑反馈（在可编辑窗口期内），系统记录修改历史；编辑超时则需教务审批回滚。
  －A3：若家长被授权查看，家长可对反馈进行回复或确认（只作为日志记录，不直接修改教师反馈）。

## Exception Flow

－E1：媒体上传失败（网络／存储异常）→ 系统提示错误并允许教师重试或先提交文字版本；失败项进入重试队列并通知技术支持。
－E2：上传媒体含敏感／违规内容（如人物隐私未授权）→ 系统触发合规检测并将反馈标记为＂待审核＂，阻止外部展示并通知教务审核。
－E3：教师无权限为该课程提交反馈（如非任课教师）→ 系统拒绝并提示需由任课教师或有权限人员操作。

## Postconditions

- 课堂反馈记录生成并关联到该节课程与学生档案。
- 相应通知／跟进任务被创建（如适用）。
- 多媒体文件保存并有访问控制与审计记录。

## Business Rules

－每条反馈应包含最小必要信息（如反馈时间、教师、课程ID、反馈类型），以便检索与统计。

- 多媒体大小／时长／格式有上限（由运维配置），超限需教师压缩或提交外链。
- 家长可见权限由教师或教务控制，且任何外放前需符合隐私同意（如拍照涉及学生须有家长授权）。
- 反馈修改需保留原始版本与修改记录，任何回滚需教务审批并记录理由。
- 可对反馈进行模板化（常用评论／评分项）以提高输入效率。
- 反馈应在规定时限内提交（例如课程结束后72小时内），过期提交需要标注＂延迟提交＂。

## Notes（实现／产品建议）

- 在教师端提供快捷输入模板与语音转文字功能，加速反馈填写。
- 支持将典型反馈内容自动生成到家长通知模版中，降低人工沟通成本。
- 为合规与存储成本考虑，可设置自动归档策略（例如一年后转冷存储）。
- 建议在反馈页面显示是否已有家长授权拍摄／上传，以避免隐私争议。

## 课程材料管理：开发难度低

## Use Case ID：UC－MAT－01

## Use Case Name

上传与分发当堂课讲义（词汇表／文档／PPT），学生可下载

## Primary Actor

－教师

## Supporting Actors

- 学生
- 系统（资料管理／文件存储、权限模块、通知中心、版本控制）
- 学导／顾问（可选接收通知与查看下载记录）

## Description

－教师在课前或课后上传当堂讲义（词汇表、文档、PPT、PDF等），系统负责存储、权限管理与分发，学生可在课后下载查看；支持版本管理与家长可见性配置。

## Trigger

－教师在课程页点击＂上传讲义＂或在课堂结束后提交材料。

## Preconditions

- 教师已登录且具有上传权限。
- 课程与该教师的授课关系已在系统登记。
- 系统具备文件存储空间与支持的文件格式列表（doc／ppt／pdf／zip／图片／音视频等）。
  －学生账户可访问对应课程资源（已enroll或有查看权限）。

## Main Flow

1．教师进入课程详情或课堂管理页面，点击＂上传讲义／教材＂。
2．教师选择或拖拽文件（词汇表、文档、PPT等），可为每个文件填写标题、说明、可见范围（仅学生／家长可见／公开）、是否允许下载、是否需要版权声明。

3．系统在上传前进行文件格式与大小校验，并提示超限或不支持的格式。
4．教师确认上传；系统将文件保存至对象存储并生成资源条目（含文件ID、上传者、时间、版本号、关联课程ID）。

5．系统在课程资源区展示新讲义，并向该课程下的学生与学导发送通知（可选邮件／推送）。

6．学生在＂我的课程 → 资源＂中查看讲义列表，并可点击下载或在线预览（若支持）。
7．系统记录下载日志（学生ID、时间、文件版本），并在教师／学导端提供下载统计与访问记录。

## Alternative Flow

- A1：教师同时上传多份文件并批量设置可见性或批量关联作业／课堂任务。
- A2：教师仅发布链接（外部资源）而非上传文件；系统存储链接并按同样权限管理。

## Exception Flow

－E1：文件超出大小或格式不支持 → 系统拒绝上传并提示压缩或转换方案。
－E2：存储服务异常或网络中断 → 系统提示失败并将上传任务放入重试队列或允许教师稍后补传。
－E3：学生无下载权限（非enroll或被限制）→ 系统拒绝下载并提示申请访问或联系学导。

## Postconditions

- 讲义文件存储并与课程关联。
- 通知已发送至目标接收人。
- 下载与访问记录被保存以供统计与审计。
- 若为更新上传，旧版本保留并生成新版本记录。

## Business Rules

- 支持的文件类型与单文件大小阈值由系统配置（超限需分片或外链）。
- 默认可见范围为＂课程内学生与教职工＂；可由教师或教务调整为家长可见或公开。
- 讲义是否允许下载为可配置属性（仅预览 vs 可下载）。
- 文件版本控制：再次上传同名讲义时系统生成新版本并保留历史版本以便回滚。
- 下载日志与访问记录必须保留 N 个月（合规由运营配置）。
- 上传含学生肖像或敏感内容时需检测并要求家长授权才能对外分发。
- 对于大文件或大量并发下载，系统应使用CDN与分片下载以保证可用性。

## Notes（实现／产品建议）

－在教师端提供＂模板库＂与一键套用讲义模板（词汇表模版、课堂PPT模板），提升效率。

- 提供在线预览（Office／PDF渲染）与移动端友好下载体验。
- 在课程资源页增加＂最近上传／最常下载／教师推荐＂排序，方便学生查找重点资料。
- 在下载记录中增加＂已读／未读＂标记与教师可见的学生阅读率统计，便于教学跟踪。

## Use Case ID：UC－VOD－01

## Use Case Name

录播课程分阶段解锁与缺课补学（含防录屏与时效控制）

## Primary Actor

－学生

## Supporting Actors

- 教师（标记进度／缺课）
- 学导（跟进）
- 教务
- 系统（学习进度引擎、权限控制、视频播放服务、DRM／防录屏模块、通知中心）

## Description

－核心产品包含配套录播课程；学生达到指定学习进度后自动解锁对应录播内容用于巩固；若班课缺课，可触发补课型录播解锁；录播需具备防录屏、防下载与有效期控制能力。

## Trigger

- 学生完成指定课程进度节点（如完成某章节／达到某课时）。
- 或教师在点名后标记学生为缺课。

## Preconditions

- 录播课程已与班课章节建立映射关系（章节ID／知识点标签）。
- 学习进度规则与缺课补学规则已配置。
- 视频平台支持DRM、动态水印、防录屏检测与播放时效控制。

## Main Flow

1．系统监听学生学习进度或出勤记录变化。
2．当学生达到预设进度节点时，系统自动解锁对应录播课程单元。

3．若学生被标记为＂缺课＂，系统根据规则自动解锁对应补课录播内容。
4．系统向学生发送解锁通知，并在＂我的课程 → 录播模块＂中展示已解锁内容与剩余有效期。

5．学生进入录播页面播放视频；系统通过授权Token校验身份并启用DRM与动态水印 （显示学生ID／手机号后几位）。

6．视频播放过程中系统监测异常行为（录屏检测、异常抓包、并发登录）。
7．解锁内容在设定有效期内可重复观看；到期后自动失效并提示续期开通或联系学导。

## Alternative Flow

- A1：教师或教务手动为特定学生提前解锁或延长有效期（例如特殊情况补课）。
- A2：学生申请补课录播，学导审核通过后系统手动触发解锁。

## Exception Flow

- E1：检测到录屏或异常抓包行为 → 系统立即停止播放并记录违规行为，通知教务。
- E2：多设备并发登录播放 → 系统限制为单设备播放并强制下线其他设备。
- E3：视频服务异常 → 系统提示稍后重试并记录播放失败日志。

## Postconditions

- 学生获得相应录播访问权限（带有效期）。
- 解锁与观看记录写入学习轨迹数据。
- 若存在违规行为，生成风控记录并通知相关人员。

## Business Rules

- 解锁规则与进度节点需可配置（按章节／课时比例／考试完成度）。
- 缺课补学解锁默认有效期（例如7天／14天）可配置。
- 视频播放必须使用DRM加密流，不提供直接下载地址。
- 强制动态水印（含用户唯一标识）以降低录屏传播风险。
- 账号默认仅允许单设备或有限设备并发播放。
- 解锁与播放行为必须可追溯（日志保存不少于N个月）。
  －若学生存在违规记录，可限制其后续解锁权限。

## Notes（实现／产品建议）

－建议使用成熟视频平台（如支持Widevine／FairPlay等DRM）的解决方案以降低开发风险。
－结合学习进度数据，推荐在录播中嵌入阶段性小测，确保学生真实学习而非仅播放。

- 在学生界面清晰显示＂解锁条件＂＂剩余观看时间＂＂违规后果＂，减少纠纷。
- 对高价值课程可增加指纹追踪（防盗链／泄露溯源）。

## 课后服务管理：开发难度中等

## Use Case ID：UC－POST－01

## Use Case Name

结课赠送写作／口语批改卡与付费延展

## Primary Actor

－学生（已结课学员）

## Supporting Actors

- 教师（批改者）
- 系统（批改卡管理、任务队列、支付模块、通知中心）
- 教务／学导（异常处理／客服）

## Description

－学生结课后系统按规则赠送若干次写作或口语批改卡（写作／口语各类或通用）。学生在系统提交作业使用批改卡；若超过赠送次数，系统引导付费购买批改次数或包月服务并继续提交批改申请。

## Trigger

－学生课程结课事件触发批改卡发放（或教师／教务手动发放）。

## Preconditions

- 学生已完成课程并在系统中标记为＂结课＂。
- 批改卡发放规则（次数、有效期、可用类型）已在规则引擎或后台配置。
- 支付模块可用，用于付费批改或购卡。

## Main Flow

1．系统在结课事件后自动发放预设数量的写作／口语批改卡到学生账户（并记录有效期与剩余次数）。

2．学生进入＂批改服务＂页面，选择＂提交写作＂或＂提交口语录音＂，填写标题／说明并上传文件（文本／音频／链接）。

3．系统校验：若学生剩余赠送批改次数 $>0$ ，则扣减一次并生成批改任务；否则提示学生购买批改包或单次付费。

4．批改任务进入教师／批改组的任务队列，教师按队列顺序进行批改并提交评分、批注与语音／文字反馈。

5．系统将批改结果推送给学生，并在学生档案／作业历史中记录批改详情、教师、耗时与状态（已完成／延期）。

6．若学生选择付费购买批改次数，完成支付后系统增加相应次数并自动提交之前待付费的批改申请（若有）。

## Alternative Flow

－A1：学生在赠送次数用尽前选择付费购买额外批改包（预付），系统先完成购买再立即生成批改任务。
－A2：教务或学导可手动为个别学生临时发放额外免费批改次数（福利／特殊情况）。

## Exception Flow

－E1：上传文件格式或大小不符合要求 → 系统拒绝提交并提示处理方式（压缩／转换）。
－E2：支付失败（网络／渠道问题）→ 订单标记为＂待支付＂，批改任务保留为＂待付费＂；系统通知学生并提供重试／客服选项。
－E3：教师超期未批改 → 系统自动提醒教师并可触发教务介入（重新分配或赔偿性补发批改卡，按规则执行）。

## Postconditions

- 批改次数／余额正确更新。
- 批改记录与反馈持久化到学生档案。
- 支付与扣次流水记录同步至财务模块。

## Business Rules

－赠送批改次数在结课时一次性发放，并有明确有效期（例如 $30 / 60 / 90$ 天，可配置）。

- 批改类型（写作／口语）与可用次数可分别配置（例如写作3次、口语2次）。
- 超次策略优先顺序：（1）使用余额／已购批改包（2）单次付费购买（3）进入人工处理 （若支付失败）。
  －批改提交格式、最大字数／时长与教师批改SLA（例如48小时内完成）需在系统显著告知并在后台强制监控。
- 支付失败或异议支持申诉与人工介入，任何退款／回退必须有教务／财务审批记录。
- 批改结果与教师信息对学生可见；教师不得泄露学生隐私，所有教师批改应保留审计记录。
  －为避免滥用，同一学生短期内批量提交可有节流规则（例如 24 小时内最多提交N次）。

## Notes（实现／产品建议）

- 在提交页显示剩余赠送次数／有效期和购卡入口，降低用户疑惑。
- 支持批改模板与快速评分项，提升教师批改效率。
- 建议在批改流程中加入反馈确认（学生可对批改结果进行满意度评价并触发质量复审流程）。
  －将批改数据统计到教学质量报表（平均批改时长、满意度、常见错题）以优化课程内容。

## Use Case ID：UC－POST－02

## Use Case Name

结课后账户延续访问（ 3 个月）

## Primary Actor

－学生（已结课学员）

## Supporting Actors

- 系统（账号权限管理、内容访问控制、通知中心）
- 教务／学导（续费与延续服务交付）

## Description

－学生结课后账户在系统中继续保留并开放一定时长（默认3个月）访问权限，允许访问指定内容（如已购课程资料、录播、批改记录等）；到期后权限回退或变更为只读／需续费。

## Trigger

－学生课程结课事件触发延续访问计时（或管理员手动设定）。

## Preconditions

－系统可区分＂结课后访问权限＂的范围（哪些资源可访问、是否可预约／购买新服务）。

- 延续期限规则（默认3个月）已配置且可调整。
- 通知机制可在到期前提醒学生续费或升级。

## Main Flow

1．课程结课后系统在学生账号上标注＂结课后访问权限＂，记录生效起始日与到期日 （默认 3 个月）。

2．在延续期内，学生可访问被允许的资源集合（示例：课程录播、讲义、课堂反馈、批改历史、作业与错题本；不包括免费预约新班课，除非另有规则）。

3．系统在学生首页或＂我的课程＂明确显示访问到期日与可访问内容说明。
4．系统在到期前（例如7天／3天／1天）发送多渠道续费提醒，提供续费／升级入口（如续保服务、购买增值包）。

5．若学生在到期前购买续期或相应服务，系统延长访问期限并同步通知。
6．到期若未续费，系统按规则对权限进行变更（例如转为只读访问、隐藏录播下载、或完全冻结账号功能），并发送到期通知。

## Alternative Flow

－A1：管理员或教务为特定学生延长访问期（如奖励、特殊处理）；系统更新到期日并通知学生。
－A2：学生在延续期内主动购买＂长期资料包＂或＂永久访问＂选项，系统立即切换权限模型。

## Exception Flow

－E1：计时或到期通知失败（消息通道异常）→ 系统重试并在失败后生成人工提醒任务给学导／客服跟进。
－E2：对资源访问计权出现异常（误判可访问范围）→ 系统进入回滚／纠错流程并记录受影响用户以人工修复。

## Postconditions

- 学生访问权限在系统中按到期／续费逻辑被正确管理并记录。
- 到期与续费操作有完整流水与审计记录。

## Business Rules

- 默认延续访问期为结课后 3 个月（可配置按课程／客户等级调整）。
- 延续期内允许访问资源范围需明确区分（阅读／预览／下载／播放／提交）并在界面中展示。
- 到期前的提醒策略（通知频率与渠道）应可配置并遵循反骚扰原则（频率限制）。
- 对 VIP／付费客户可默认延长或提供优惠续期政策。
- 任何延长或提前停用权限的人工操作必须记录操作者与理由并可审计。

## Notes（实现／产品建议）

－在学生端清晰展示＂结课后可用功能清单＂以避免误解（例如哪些功能继续可用，哪些需要另付费）。
－将＂结课后访问＂作为可售产品打包（续期／资料包／批改包），形成后续变现路径。
－在到期前自动生成运营线索供顾问跟进（如高活跃但未续费用户）。

## 新生Onboarding流程管理：开发难度高

## Use Case ID：UC－OC－01

## Use Case Name

新生OC流程（表单填写→学导分派 → 开课仪式 → 面谈建档 → 排课需求 → VIP家长反馈）

## Primary Actor

- 顾问（OC表填写）
- 学导主管（分派）
- 学导（面谈与反馈）
- 学生／家长

## Supporting Actors

- 教务
- 教学负责人
- 系统（CRM、任务分配、电子签字、学生档案、排课需求、通知中心、审计模块）

## Description

－新生从顾问填写OC表开始，经过学导分派、OC开课仪式、面谈建档与排课需求生成，最终完成入学与后续教学准备；VIP学生需额外提交家长反馈并由教学管理层检查。

## Trigger

－顾问在系统提交OC表。

## Preconditions

－学生基础档案已存在或可在OC表中创建。

- 系统配置学导分派规则、电子签字流程与面谈模板。
- VIP学生有标签标识。

## Main Flow

1．顾问填写并提交OC表（基本信息、课程意向、优先级等）。
2．系统生成待分派任务并通知学导主管。
3．学导主管按规则（校区／负荷／VIP优先）分派责任学导。
4．系统通知责任学导并更新学生归属。
5．学导安排并发起OC开课仪式流程。
6．系统展示开课须知与规则，学生／家长完成电子签字确认。
7．系统将学生状态更新为＂已开课＂。
8．学导进行面谈并在系统填写完整学生档案：

- 基本信息（年龄、性别、院校）
- 真考成绩、测评结果、词汇量
- 目标成绩、预计真考时间
- 是否报名真考及报考时间
- 性格爱好与学习偏好

9．学导根据面谈结果填写排课需求表（科目、课时、频率、时间窗、最迟完成期限）。

10．系统保存档案并生成排课需求任务，通知教务。
11．若学生为VIP：

- 学导在系统提交家长反馈（开课说明、学习规划、注意事项）
- 教务或教学负责人在线查看并标记已阅或退回补充。

1． OC 流程完成，进入排课与正式授课阶段。

## Alternative Flow

- A1：自动分派失败 → 学导主管手动分派。
- A2：电子签字无法在线完成 → 线下签署并上传确认。
- A3：学生面谈后信息缺失 → 系统标记为＂待补充＂，不阻断排课但提醒学导完善。
  －A4：VIP反馈被退回 → 学导修改后重新提交。

## Exception Flow

- E1：OC表信息不完整 → 系统拒绝分派并要求补充。
- E2：无可用学导 → 系统进入待分派队列并通知主管。
- E3：电子签字异常 → 标记为＂待签署＂，不得进入正式排课。
- E4：面谈数据冲突或异常 → 系统记录版本并提示人工核验。

## Postconditions

- 学生成功完成OC流程并进入排课阶段。
- 学生档案与排课需求完整存档。
- VIP学生家长反馈完成并可追溯。
- 全流程操作均留存审计记录。

## Business Rules

- OC未完成（未签字）不得进入正式排课。
- 分派规则可配置（校区优先、负荷均衡、VIP优先）。
- 档案修改必须保留历史版本。
- VIP反馈需在开课后X天内提交。
- 所有关键步骤需可追溯（签字、分派、档案修改、反馈提交）。

学生类型划分与学习规划生成（常规／VIP）
Primary Actor

## Description

- 常规学生基于统一学习规划模板自动生成《阶段学习任务表》。
- VIP学生由学导自定义生成《个人专属学习规划》，支持按日细化任务。
- 系统统一跟踪学习任务完成进度，并在所有相关端口可见。

## Trigger

- 学生完成OC流程并进入正式学习阶段。
- 或学导在学生档案中设置／变更学生类型标签。

## Preconditions

- 系统中已配置＂常规学习规划模板＂（按科目／阶段）。
- 学生已被标记为常规或VIP。
- 任务管理与进度跟踪模块可用。

## Main Flow（常规学生）

1．学导在学生档案中确认学生类型为＂常规学生＂。
2．系统根据课程类型与阶段自动匹配统一模板。
3．系统生成《阶段学习任务表》，包含听、说、读、写各科的练习内容与基础要求 （例如每日词汇量、每周阅读篇数、口语练习频次等）。

4．系统将任务按时间维度分配（例如按周划分阶段任务）。
5．学生在系统中查看任务列表并逐项提交完成情况（打卡／上传作业／系统自动统计）。

6．系统自动计算完成率并生成阶段进度百分比。
7．学导、教务、教学负责人在各自端口可查看学生任务进度与达成情况。

## Main Flow（VIP学生）

1．学导在学生档案中将学生标记为＂VIP学生＂。
2．学导进入＂个人专属学习规划＂编辑界面。
3．学导根据学生情况自定义学习目标与阶段任务（可复制模板后调整）。
4．系统支持按日或按时间段发布任务（例如：次日9：00－10：00词汇，14：00－15：00阅读）。

5．系统提前生成并发布次日详细任务安排。
6．学生按时间段完成任务并在系统打卡或提交成果。
7．系统实时更新任务完成率与整体进度曲线。
8．所有相关角色（学导、教务、教学负责人）可在后台查看VIP学生的详细执行情况。

## Alternative Flow

- A1：学导在学习过程中调整任务目标或频率，系统记录版本变更。
- A2：学生申请延长阶段时间，学导审核后修改阶段截止日期。
- A3：学生类型发生变更（常规 → VIP或VIP → 常规），系统保留原规划历史并生成新规划版本。

## Exception Flow

- E1：学生未按时完成任务，系统触发提醒通知给学生与学导。
- E2：数据未同步或任务记录异常，系统标记为＂数据异常＂并通知技术或教务。
- E3：学导误删除或覆盖规划，系统支持版本回滚。

## Postconditions

- 学生拥有明确的阶段或个人学习规划。
- 任务完成情况可视化展示。
- 所有端口（学生、学导、教务、教学负责人）均可查看进度。
- 所有任务变更与完成记录存档。

## Business Rules

- 学生类型标签必须唯一（常规或VIP）。
- 常规学生规划基于统一模板生成，禁止随意修改核心结构（仅允许少量调整）。
- VIP规划允许完全自定义，但必须设定明确阶段目标与截止时间。
- 系统必须展示阶段完成率（完成任务数／总任务数）。
- 任务完成记录不可删除，仅可标记作废并保留历史。
- 规划版本变更需保留版本号与修改人。
- 若连续 N 天未完成任务，可触发预警并上报学导。

## Notes（实现／产品建议）

- 建议在系统中设计＂学习进度仪表盘＂，展示进度条、完成率、连续打卡天数。
- 常规学生以模板驱动为主，降低管理成本；VIP学生以精细化管理为主，支持个性化配置。
- 可引入＂智能建议任务量＂机制，根据学生完成率自动建议调整强度。
- 为VIP学生增加家长可见模块（可选），提升家长参与度。

## Use Case Name

基于开课资料的学习任务发布与进度跟踪

## Primary Actor

－系统（任务管理／内容关联模块）

## Supporting Actors

- 学生
- 教师
- 学导
- 系统（资源管理、进度追踪、通知中心）

## Description

－课程创建并上传配套学习资料，或教务／教师在课程设置中启用＂指定任务＂功能。

## Preconditions

- 课程资源（精听、精读等）已上传并按章节／等级标注。
- 系统可配置任务模板与默认完成期限。
- 学生已被enroll到课程或被授权访问资源。

## Main Flow

1．教师／系统为课程选择或自动生成任务清单，并为每项任务设置完成截止时间（可按学生或班级区分）。

## Exception Flow

－E1：资源缺失或格式问题导致学生无法完成任务 → 系统标注任务异常并通知教师补充资源。
－E2：学生离线提交（如上传失败）→ 系统进入重试队列并提示学生／教师。

- 任务完成记录与进度统计被写入学生学情档案。

- 任务完成以系统可验证的提交行为为准（打卡／上传／自动判分），手工标记需有审批流。
  －系统应支持按资源等级（A／B／C）汇总完成率与难度加权进度。

## Notes（实现／产品建议）

- 在学生端显示预计完成时间线与每日建议量，提升完成率。
- 提供任务完成漏斗与趋势图，便于教师在上课前快速诊断班级整体准备度。

## Use Case ID：UC－LEARN－02

## Use Case Name

基于错题自动生成生词本与错题本

## Primary Actor

## Supporting Actors

- 学生
- 教师
  －系统（OCR／题型解析／标签化服务）

## Description

－系统根据学生做题或提交作业中的错题自动抽取生词并生成个人生词本；生词本支持复习计划与导出，供学生和教师查看与推送个性练习。

## Trigger

－学生完成测验／作业或上传拍照题目并被判为错误（自动判分或教师标注错题）。

## Preconditions

－题库／作业与知识点映射已建立；系统支持从题目中抽取词汇（OCR 与 NLP 支撑）。

2．系统对错题进行词汇抽取与词性识别，生成候选生词条（词形、中文释义、例句、关联知识点）。

- A1：学生选择手动将某些词加入个人生词本（非错题触发）。
- A2：教师从班级错题池批量提取常见生词并下发至所有学生生词本。

## Exception Flow

－E1：OCR／NLP无法准确识别题目或词汇 → 系统将该错题标注为＂需人工校对＂，并推送给教师或学生确认。
－E2：同词多个变形或不同语境导致冲突 → 系统保留多条候选并允许人工合并。

## Postconditions

- 个人生词本更新并与学习任务／复习计划联通。
- 生词使用与掌握数据计入学情数据供后续智能推题使用。

## Business Rules

- 生词自动入库需有置信度阈值（低置信度需人工确认）。
- 同一生词在学生生词本中去重并统计错题出现次数以作为优先复习权重。
- 生词本项应支持多媒体（发音、例句、音频）以提升复习效果。
- 导出或与第三方同步需遵守数据权限与隐私规则。

## Notes（实现／产品建议）

- 建议把错题池与生词本做联动展示，教师可基于高频错词设计专门讲义。
- 对于语音／口语错题，尝试自动提取关键词并纳入生词本。

## 模拟考试结果分析（AI）：开发难度高

## Use Case ID：UC－MOCK－01

## Use Case Name

模考计划设置、成绩流转与模考报告生成（外置考试链接）

## Primary Actor

- 学导（模考计划管理）
- 学生（参加模考）

## Supporting Actors

- 任课教师（评分与审核）
- 系统（模考计划模块、提醒模块、成绩管理模块、Al评分模块、报告生成模块、通知中心）
  －家长（查看报告）

## Description

- 根据学生真考时间与课程进度设置阶段性模考计划；
- 学生通过外置链接完成模考练习；
- 成绩回传至系统进行评分与分析（听力阅读可自动评分，口语写作AI＋人工复核）；
- 系统生成模考报告并经学导／教师审核后发布给学生与家长查看。

## Trigger

- 学导在学生档案中设置或更新＂真考时间＂。
- 或学生进入某课程阶段节点（阶段内／结课后）。

## Preconditions

- 学生档案包含预计真考时间。
- 模考阶段规则已配置（每阶段模考次数、提醒时间、报告模板）。
- 外置模考平台可回传成绩或支持手动录入成绩。

## Main Flow

## 一、模考时间规划与提醒

1．学导在系统中录入或确认学生真考时间。
2．系统根据真考时间与课程阶段自动生成模考时间点（例如：阶段中1次、阶段末1次、结课后冲刺1次）。

3．每个阶段设定模考次数与建议时间窗口。
4．在模考时间点前X天，系统提醒学导与学生预约线下模考时间（或确认外置考试时间）。

5．学导在系统中确认模考安排时间并生成提醒任务。

## 二、模考执行与成绩录入

1．学生在外置模考平台完成听力、阅读、口语、写作考试。
2．成绩通过接口回传至系统，或由学导／教务手动录入分数。
3．听力与阅读成绩直接记录为自动评分结果。
4．口语与写作成绩流程：

- 系统可先调用AI进行初步评分与评语生成；
- 将结果推送给任课老师进行人工审核与修订；
  －在规定时间内完成最终评分确认。
  5．系统记录评分时间与评分人，若超过SLA自动提醒教师。

## 三、模考报告生成与审核发布

1．系统根据模考成绩生成模考报告草稿，包括：

- 总分
- 对应欧标分数换算
- 听说读写各单科分数
- 单科成绩趋势
- 题型弱点分析
- 高频错误分析
- 后续学习建议

1．报告生成后进入＂待审核＂状态。
2．学导与任课老师可在线查看报告并补充分析或修改建议。
3．审核完成后点击＂发布报告＂。
4．系统向学生与家长发送通知，开放报告查看权限。
5．模考数据同步进入学生成长档案与趋势图。

## Alternative Flow

- A1：外置平台不支持自动回传成绩 → 由教务人工录入成绩后触发报告生成。
- A2：教师认为AI评分偏差过大 → 完全人工重评并覆盖AI结果。
- A3：学导可调整模考时间计划（如考试时间提前／延后）。

## Exception Flow

- E1：外置考试成绩未成功回传 → 系统生成＂待录入成绩＂任务并通知教务。
- E2：教师未在规定时间完成评分 → 系统自动提醒并升级通知教学负责人。
- E3：报告审核未通过 → 退回编辑并记录修改历史。

## Postconditions

- 模考成绩记录完整存档。
- 模考报告生成并发布。
- 数据进入学生成长趋势分析。
- 审核与修改记录可追溯。

## Business Rules

- 模考次数与阶段配置必须可配置（不同课程不同规则）。
- 报告必须经人工审核后方可对学生端开放。
- AI评分仅作为辅助，最终成绩以人工审核为准。
- 成绩与报告必须长期保存，支持历史对比。
- 欧标换算规则需标准化并锁定版本。
- 提醒规则（提前X天、评分SLA）必须可配置。

## Notes（设计优化建议）

- 建议增加＂模考趋势图＂，展示多次模考分数变化曲线。
- 可对弱项自动推荐对应训练模块，形成闭环。
- 对VIP学生可增加＂模考家长解读会＂预约入口。

## 学习报告汇总（AI生成）：开发难度高

## Use Case ID：UC－REPORT－02

## Use Case Name

阶段性学习报告（期中／期末）：自动汇总 + 人工寄语

## Primary Actor

－系统（报表引擎／学情模块）

## Supporting Actors

- 任课教师
- 学导（学业顾问）
- 学生
- 家长
- 教务／教学负责人

## Description

－在每个学习阶段的期中与期末，系统自动生成学生学习报告草稿（覆盖已学内容进度、单词量进度、作业完成率、阶段内模考成绩等），任课教师和学导可在线补充人工寄语与建议；报告经审核后发布给学生与家长并归档。

## Trigger

－到达阶段性节点（期中／期末／自定义阶段结束）或教务手动触发生成报告。

## Preconditions

- 课程进度、出勤、作业提交、模考成绩、词汇／测评数据等已同步至学情模块。
- 报告模板与所需字段在系统中已配置。
- 任课教师与学导有编辑／审核权限。

## Main Flow

1．系统在阶段结束触发报表生成任务，自动聚合：已教学内容覆盖率、学生单词量增长、作业完成率与质量统计、阶段内模考成绩与趋势、错题／弱项概览。

2．系统生成报告草稿并在教师／学导端创建待填写草稿项（包含可编辑的＂寄语／建议＂字段）。

3．任课教师在教师端登录查看草稿、补充教学评价与寄语（文字／评分／可附文件）。
4．学导在学导端查看并可补充家校沟通角度的寄语或运营建议（可编辑或直接通过接受）。

5．教师或学导完成填写后将报告提交审核（可多轮），教学负责人可审核并退回修改。

6．审核通过后，系统将报告以页面与可导出 PDF 的形式发布到学生／家长端，并发送通知（站内推送 + 邮件／短信可选）。

7．报告被归档到学生档案，作为历史学情与考评依据；各端口可检索历史报告并生成趋势图。

## Alternative Flow

－A1：教师暂保存草稿并在截止前继续完善 → 系统在截止日提醒未完成报告的教师／学导。
－A2：对于VIP或特殊学生，学导／教务可申请延长填写期限或组织线上家长解读会并在报告中附录会议纪要。

## Exception Flow

－E1：缺失关键数据（如模考成绩未回传）→ 报告进入＂待补全数据＂状态并生成数据补录任务通知相关人员；可选择先发布部分内容并标注未完备项。

- E2：审核超时或责任人未处理 → 系统自动升级提醒并抬升到教学负责人待办列表。
- E3：数据异议（学生／家长对报告内容提出申诉）→ 进入申诉与复核流程，必要时修订报告并保留版本记录。

## Postconditions

- 阶段性报告发布给学生与家长（或保持草稿／待发布状态）。
- 报告与教师／学导寄语、审核记录、版本历史被保存入学生档案。
- 报表关键指标写入BI以便跨学生／班级／校区汇总分析。

## Business Rules

－报告模板字段为必填项（自动汇总部分与人工填写部分需区分并强制填写最小项）。

- 报告发布前必须通过至少一名任课教师或学导的确认（具体规则可配置）。
- 报告版本不可删除，仅可追加修订并保留历史版本与修改人。
- 报告中涉及成绩或评语的敏感字段仅对有权限的角色可见（家长／学生可见范围由家长同意与隐私策略控制）。
  －报告生成与教师／学导填写有 SLA（例如生成后7日内完成初稿，审核48小时内完成），逾期触发升级提醒。
  －系统应支持批量生成并分配报告任务以提升运营效率。

## Notes（实现／产品建议）

－建议自动汇总区域用图表（进度条、词汇增长曲线、作业完成率饼图、模考分数趋势）＋人工寄语文本区分展示，便于阅读。

- 提供＂家长解读模板＂供学导在发布后一键发送解读要点或预约家长会。
- 支持导出 PDF 与打印，并在导出中嵌入签名／确认区以便家校签收。
- 为便于质量控制，引入报告质量评分机制（家长／学生满意度反馈）并纳入教师绩效统计。

## 真考成绩记录（AI生成评论与改进建议）：开发难度高

## Use Case ID：UC－EXAM－01

## Use Case Name

真考成绩录入与证据上传

## Primary Actor

－学导／顾问／教务

## Supporting Actors

- 学生（提供成绩截图）
- 系统（成绩管理模块、文件存储、OCR／验证服务、通知中心）
- 教师（可查看成绩记录）

## Description

－负责人员在系统中录入学生真考成绩并可上传成绩截图／证明；系统保存成绩源文件并可对图片做OCR／元数据校验以便后续审核与归档。

## Trigger

－学生或顾问收到真考成绩并提交证明；或教务主动录入成绩。

## Preconditions

- 学生档案存在且已关联该考试记录（考试类型／日期等）。
- 文件存储／附件模块可用。
- 系统配置允许上传的文件类型与大小限制。

## Main Flow

1．责任人进入学生考试记录页面并选择＂录入真考成绩＂。
2．填写成绩字段（总分、单科分数、考试日期、考试类型、考场／报名号等）。
3．上传成绩截图或证明文件（图片／PDF）；系统提示支持格式与大小。
4．系统保存成绩数据并把原始文件存储到对象存储，生成附件ID与校验元数据（上传时间、上传人、文件哈希）。

5．系统可选触发OCR识别并提取关键字段作为辅助校验（例如总分／单科）；若识别结果与手工录入不符，系统标注异常并提示复核。

6．系统将成绩记录标记为＂已录入／待审核＂并通知任课教师与学导（如需人工复核）。
7．审核通过后成绩状态变为＂已确认＂，写入学生成长档案与报告计算模块。

## Alternative Flow

- A1：上传文件不合格（格式／大小）→ 系统拒绝并提示上传规范。
- A2：学生通过外部链接提交成绩（如考试官网），教务可粘贴链接并标注为证据，系统存储链接并做回溯记录。

## Exception Flow

－E1：OCR识别失败或结果低置信度 → 系统将该记录标注为＂需人工核对＂，并推送给教务／学导人工复核。
－E2：上传文件涉嫌伪造或不一致 → 系统生成风控工单并通知教学负责人人工处理。

## Postconditions

－真考成绩与原始证明被持久化并可追溯。

- 学生档案中更新真考成绩字段，触发后续报告／计划生成流程。
- 审核与变更记录保存在审计日志中。

## Business Rules

- 支持文件类型：jpg／png／pdf；单文件大小上限由系统配置（默认示例：10MB）。
- 上传文件必须保留原始哈希与时间戳以便取证。
- OCR仅作辅助；人工录入或复核为最终依据。
- 成绩一旦＂已确认＂后修改需记录修改人、理由并保留历史版本。
- 敏感数据访问受权限控制，只有有权角色可查看原始成绩截图。
- 若成绩与合同／退款策略相关，需在财务接口上触发相应业务（如退款／奖学金判定）。

## Use Case ID：UC－EXAM－02

## Use Case Name

不达标智能诊断与真考后学习计划（AI建议 → 教师调整 $\rightarrow$ 真考报告单）

## Primary Actor

- 系统AI引擎（自动建议）
- 学导／任课教师（调整与审核）

## Supporting Actors

- 学生（目标分／反馈）
- 系统（学情分析模块、任务规划模块、报告生成模块、通知中心）
- 财务／运营（如触发补课／付费服务）

## Description

－系统根据录入的真考成绩与事先设定的目标分判断是否达标；若未达标，AI根据学生历次表现（单科分数、模考／作业／错题、出勤、词汇量等）自动生成各科建议课时与学习计划草案；教师／学导在系统中可调整建议并最终生成正式真考报告单供发布给学生与家长。

## Trigger

－录入并确认真考成绩；或学生在系统查看成绩触发＂生成真考建议＂。

## Preconditions

- 已录入并确认的真考成绩可用。
- 学生的历史学情数据（模考、作业、错题、本阶段进度、词汇量、出勤等）已同步至学情模块。
- 系统已配置目标分基线及可用的AI评分／建议模型。
- 教师／学导具有编辑与审核权限。

## Main Flow

1．系统读取学生目标分（先前设定）并比对真考实际分；判定是否达标。
2．若未达标，系统触发 AI 诊断模块：聚合近期模考、作业表现、错题分布、单科低分点、词汇量缺口、出勤与任务完成度等指标。

3．AI基于诊断结果计算每个科目的建议课时量（例如：听力 +8 课时、写作 +12 课时等），并生成一个包含优先级的＂后续学习计划草案＂（阶段目标、推荐练习／资料、评估节点与预计完成时长）。

4．系统将AI草案推送给任课教师与学导以供查看和修改；同时自动生成预估成本与时间表（用于营收／续班建议）。

5．教师／学导在系统中审核并可调整：修改课时建议、添加具体作业或指定教师、设定复测节点或优惠策略。

6．审核完成后，系统生成最终＂真考报告单＂（含真考成绩、目标对比、单科分析、题型弱点、建议课时、学习计划、教师／学导寄语、下一步建议与可能产品／服务报价）。

7．报告单由教师／学导签字确认后发布给学生与家长；同时生成跟进任务（如续班沟通、预约诊断课、购买课时包等）。

8．系统将该计划纳入任务管理模块并开始跟踪执行（任务完成情况反馈至学情模块）。

## Alternative Flow

－A1：学生接受AI建议并直接线上购买推荐课时包，系统自动安排优先入班并更新计划执行状态。
－A2：教师认为AI建议不合适 → 直接手动制订学习方案并覆盖AI草案；系统记录人工决策原因。

## Exception Flow

－E1：学情数据不足或数据冲突导致AI诊断不可靠 → 系统标注＂建议为低置信度＂，并要求教师人工诊断后生效。
－E2：教师长时间未审核AI草案 → 系统发送催审提醒并将该任务升为高优先级或抬送至教学负责人。
－E3：学生对建议提出异议 → 进入申诉／复议流程，由学导与教师协调并记录结果。

## Postconditions

- 若未达标，生成并发布经过教师审核的真考报告单与学习计划。
- 系统记录AI草案、教师修改历史与最终批准记录以供审计与模型回溯。
- 跟进任务（课时购买、补课安排、复测计划）已生成并进入执行流程。

## Business Rules

- AI建议仅为辅助决策，最终课时与计划需教师／学导审核确认方可生效。
- 建议课时量与定价策略需与运营／财务规则联动（例如折扣、续费优先级）。
- 若AI置信度低或基础数据不足，系统不得自动下发计划，必须人工干预。
- 报告单发布前必须记录审核人、审核时间与修改摘要。
- 任何涉及收费的后续建议在生成前须提示学生／家长并取得支付确认。
- AI模型与建议策略需定期评估其校准效果并保留版本记录用于合规审计。
- 敏感成绩与诊断结果的访问受权限与隐私政策保护，仅对授权角色／家长可见。

## Notes（实现／产品建议）

－建议在报告中同时呈现＂AI依据要点＂（例如：低分源于听力题型A与词汇缺失），提高透明度并便于教师调整。

- 保留AI草案与最终报告的差异视图，便于后续优化模型与教师培训。
- 对高价值／VIP学生可引入专属人工诊断流程（人工优先于 AI ）。
- 将该流程与财务／销售流程打通，形成从诊断 → 推荐 → 成交 → 教学执行的闭环。

## Use Case ID：UC－FOLLOW－01

## Use Case Name

常规／VIP 学导差异化沟通反馈与进度监管

## Primary Actor

－学导（常规学导／VIP学导）

## Supporting Actors

- 学导主管
- 教务／教学负责人
- 顾问
- 系统（任务提醒模块、沟通记录模块、权限与审计模块、通知中心）

## Description

－常规学导与VIP学导有不同的沟通反馈频率与内容要求；系统按学生类型自动生成周期性沟通任务并在规定时间内提醒学导完成；学导将沟通记录写入系统，其他相关部门可查看，主管可监控完成进度与质量。

## Trigger

- 学生被分配至学导并标记为＂常规＂或＂VIP＂。
- 或到达预设沟通周期节点（例如每周／每两周／每月）。

## Preconditions

- 学生类型（常规／VIP）已在系统中标识。
- 不同类型的沟通频率与反馈模板已在系统中配置。
- 学导账号具备填写反馈权限，主管具备查看与统计权限。

## Main Flow

1．系统根据学生类型自动生成沟通计划：
－常规学生：例如每两周一次学习反馈；
－VIP学生：例如每周一次详细反馈 + 阶段总结。
2．系统在沟通截止日前X天提醒学导（站内提醒／消息推送）。
3．若到截止日未完成，系统再次提醒并标记为＂逾期＂。
4．学导在学生档案中填写沟通反馈内容，包括：

- 沟通方式（电话／微信／面谈）
- 沟通对象（学生／家长）
- 学习进展摘要
- 问题与建议
- 后续行动计划

5．学导提交反馈，系统记录提交时间与内容版本。
6．反馈自动同步至学生档案与沟通历史模块。
7．其他部门（教务／顾问／教学负责人）可查看该沟通记录（只读权限）。
8．学导主管在＂工作看板＂中查看：

- 各学导沟通完成率
- 逾期记录
- VIP沟通完成情况
- 阶段汇总统计。

## Alternative Flow

- A1：学导提前完成沟通并填写反馈 → 系统标记为＂已完成＂。
- A2：学导因特殊情况申请延期 → 主管审核后延长截止时间。
- A3：对VIP学生可增加额外专项反馈（如模考后专项沟通）。

## Exception Flow

- E1：学导长期未完成反馈 → 系统自动升级提醒给主管。
- E2：沟通记录填写不完整（缺少必填字段）→ 系统拒绝提交并提示补充。
- E3：权限异常（无权限查看他人学生反馈）→ 系统拒绝访问并记录日志。

## Postconditions

- 沟通记录写入系统并归档。
- 完成率与逾期数据更新到主管工作看板。
- 所有操作记录保存在审计日志中。

## Business Rules

- 常规与VIP沟通频率可配置（例如常规每2周，VIP每周）。
- VIP反馈需包含更多必填项（如阶段目标完成率、家长满意度）。
- 逾期规则需可配置（例如逾期3天自动升级）。
- 沟通记录不可删除，仅可补充修改并保留历史版本。
- 主管可查看所有学导的反馈数据，但不可随意修改他人记录。
- 所有沟通记录对相关部门可见，但需遵守数据权限控制。

## Notes（实现／产品建议）

- 建议设计＂学导工作仪表盘＂，显示待完成沟通数量、逾期提醒、VIP学生比例。
- 对VIP沟通可支持家长在线确认或评价。
- 将沟通完成率纳入绩效考核模块。
- 可生成月度沟通统计报表供管理层查看。

# Use Case ID: UC-PROFILE-01

## Use Case Name

学生全生命周期档案管理与分析

## Primary Actor

- 系统

## Supporting Actors

- 教师
- 学导
- 学生
- 家长
- 教务

## Description

- 系统为每个学生建立完整档案，涵盖基础信息、学习行为、作业表现、模考成绩、在线学习数据，并支持阶段性汇总与真考后的报告生成。

## Trigger

- 学生完成OC入学或首次进入系统。

## Preconditions

- 学生已完成档案创建流程（OC/面谈）。
- 学习数据（作业、模考、在线行为）已接入系统。

## Main Flow

1. 系统自动生成学生基础档案（基本信息、背景信息）。
2. 系统记录学生课前学习情况（预习、学习资料完成情况）。
3. 系统持续记录课堂与课后数据：
    - 作业完成情况
    - 学习任务完成情况
    - 教师反馈
4. 系统记录模考数据并生成模考报告。
5. 系统收集在线学习行为数据：
    - 在线时长
    - 作业完成率
    - 正确率排名
6. 系统按阶段生成汇总数据（期中 / 期末）。
7. 教师或学导可对档案进行补充修改（如备注、评价）。
8. 当真考成绩录入后：
    - 若达标 → 生成结课报告
    - 若未达标 → 生成后续学习建议报告

## Alternative Flow

- A1: 教师手动补充分析或评价。
- A2: 学生数据缺失时，系统标记为“数据不完整”。

## Exception Flow

- E1: 数据同步失败 → 系统标记异常并通知管理员。
- E2: 成绩或行为数据冲突 → 进入人工校验流程。

## Postconditions

- 学生形成完整可追溯学习档案。
- 数据可用于报告生成与教学决策。

## Business Rules

- 所有数据需按时间轴记录，不可覆盖历史。
- 阶段报告必须基于系统数据自动生成。
- 教师修改需记录操作日志。
- 真考结果必须触发报告生成逻辑。

# Use Case ID: UC-COMM-01

## Use Case Name

家校沟通记录与进度透明化管理

## Primary Actor

- 学导

## Supporting Actors

- 教师
- 顾问
- 教务
- 主管
- 系统（沟通记录模块、权限模块）

## Description

- 所有与学生及家长的沟通（微信、电话、面谈等）统一记录在系统中，实现跨部门可见与主管进度监管。

## Trigger

- 学导或教师与学生/家长发生沟通行为。

## Preconditions

- 学生档案已存在。
- 用户具备沟通记录权限。

## Main Flow

1. 学导或教师在系统中创建沟通记录。
2. 填写沟通信息：
    - 沟通方式（微信/电话/面谈等）
    - 沟通对象（学生/家长）
    - 沟通内容摘要
    - 后续跟进事项
3. 系统保存记录并关联学生档案。
4. 系统将记录开放给相关角色查看（只读）。
5. 主管在后台查看：
    - 沟通频率
    - 沟通内容
    - 跟进状态
6. 系统可按时间线展示所有沟通历史。

## Alternative Flow

- A1: 支持补录历史沟通记录。
- A2: 支持附件上传（如聊天截图、录音）。

## Exception Flow

- E1: 记录不完整 → 系统拒绝提交并提示补充。
- E2: 权限不足 → 拒绝查看并记录访问行为。

## Postconditions

- 所有沟通行为被系统化记录。
- 跨部门信息透明。
- 主管可实时掌握学生跟进进度。

## Business Rules

- 沟通记录不可删除，仅可补充修改。
- 所有记录需带时间戳与操作人。
- 不同角色查看权限需控制（隐私保护）。
- 沟通记录需支持审计与导出。

# Use Case ID: UC-ADMIN-01

## Use Case Name

账户与角色权限管理（RBAC）

## Primary Actor

- 系统管理员（Admin）

## Supporting Actors

- 教务
- 学导
- 教师
- 顾问
- 系统（权限控制模块）

## Description

- 管理员创建、编辑用户账户，并基于角色分配系统权限，实现不同岗位访问控制。

## Trigger

- 新员工入职或岗位调整。

## Preconditions

- 系统已定义角色模型（教务、教师、学导、顾问等）。
- 管理员具备最高权限。

## Main Flow

1. 管理员进入“账户管理”模块。
2. 创建新用户账户（姓名、手机号、邮箱、所属校区）。
3. 分配角色（如教务/学导/教师）。
4. 系统自动加载对应权限模板。
5. 管理员可对权限进行细粒度调整（模块级/功能级）。
6. 保存并激活账户。
7. 系统通知用户账户已创建。

## Alternative Flow

- A1: 批量导入账户（Excel）。
- A2: 临时权限授权（如跨校区访问）。

## Exception Flow

- E1: 重复账号 → 系统拒绝创建。
- E2: 权限冲突 → 系统提示修正。

## Postconditions

- 用户账户可登录系统。
- 权限生效。

## Business Rules

- 权限遵循最小权限原则。
- 所有权限变更需记录日志。
- 高权限操作需二次确认。
