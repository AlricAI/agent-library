# DAO Recovery Candidates

> Date: 2026-03-20
> Scope: legacy DAO/platform briefs, old repos, archived launch packages
> Status: recovery library only, not live truth

## Recovery Rule

Legacy DAO repos, archived briefs, and abandoned launch packages may be reused only as a recovery library.

They do **not** count as live implementation, prior customer revenue, or completed infrastructure unless the work is:

1. ported into `C:\ANTIGRAVITY`
2. validated in the current repo
3. documented as current truth in live briefings
4. kept inside the one-repo `main` baseline

If an old brief says something is "complete" or "live" but it is not present in the current repo/runtime, treat it as design material only.

These are beneficial to the long-term mission and safe to reintroduce intentionally:

1. **Visual systems**
   - Dating palette and motion direction for YouAndINotAI
   - Marketplace palette and card system for AI-Solutions.Store
   - Command-center visual language for ClawX/admin surfaces

2. **Workspace packaging**
   - Two separate downloadable workspace exports
   - Clean split between dating product and marketplace product
   - Shared control-plane workspace as the fifth project bucket

3. **DAO/governance documentation**
   - governance thresholds
   - tokenomics drafts
   - treasury logic notes
   - proposal taxonomy

4. **Operator tooling patterns**
   - admin dashboard layout ideas
   - agent-role decomposition
   - non-customer-facing monitoring/reporting concepts

5. **Passive-income support concepts**
   - content automation patterns
   - media pipeline ideas
   - packaging/fulfillment automation

## Not Safe To Recover Blindly

These stay out unless rebuilt and validated in the live repo:

1. **Claims of prior completion**
   - self-hosted K8s completion
   - full DAO launch completion
   - agent mesh completion
   - payment readiness claims

2. **Mixed product/runtime surfaces**
   - one merged app for dating plus marketplace
   - shared payment logic between products
   - shared customer data models across products
   - shared compliance assumptions

3. **Stale infra assumptions**
   - old domains
   - old cloud topology
   - dead node layouts
   - old GitOps/bootstrap scripts presented as current

4. **Unproven revenue rails**
   - old staking/disbursement claims
   - old webhook/payment completion assumptions
   - simulated or guessed business metrics

## Current Call

The mission-beneficial recovery path is:

1. keep **one live repo** and **one live truth**
2. treat old DAO/platform work as a design archive
3. selectively re-import only high-signal parts that strengthen durability
4. prefer design systems, workspace structure, and governance docs before infra sprawl

## Mission Guardrail

If a recovered item improves resilience, clarity, governance, treasury safety, or long-term continuity for the kids **without** increasing launch risk, it is a candidate.

If it adds drift, fake readiness, mixed product boundaries, or deploy complexity, it stays archived.
